# capa_paymentguard

A showcase of [Capa](https://github.com/nelsonduarte/capa-language)'s
**capability discipline** and **information-flow control** applied to a
payment-transaction security core, modelled on the requirements of
PCI DSS v4.0.1, PSD2 / SCA, ISO/IEC 27001, RGPD and OWASP.

It is built to demonstrate the one property no mainstream language gives
you for free:

> **the compiler proves that a card number can never reach a log line or
> a network call unless it is explicitly masked, and records every such
> disclosure, with its reason, in the SBOM.**

## What this is, and what it is NOT

This is a **language showcase**, not a production payment system. Read
the honesty section below before drawing any other conclusion.

### The strong claims (these are real)

- **Information-flow control.** Card data (`pan`, `cvv`) and credentials
  are typed `@secret`. The compiler propagates that label through every
  derived value and reports, at compile time, any `@secret` value that
  reaches a public sink (`Stdio.println`, `Fs.write`, `Net.post`, ...).
  See `leaky_example.capa` for the violation being caught, and
  `mask.capa` for the only sanctioned way across for cardholder data.
- **Least privilege, proven.** The fraud / AML engine (`fraud.capa`)
  takes **no capabilities at all**. The SBOM proves it cannot touch the
  network, the filesystem, the clock, or the environment. `main` holds
  only `Stdio` and `Fs`, and `Net`/`Proc`/`Db`/`Unsafe` appear nowhere.
- **Auditable disclosure.** Every secret-to-public bridge is a
  `declassify(value, reason: "...")` call. The reasons are emitted into
  the SBOM as `declassification_sites`: a machine-checkable record of
  where, and why, the program discloses cardholder data (see
  `conformity/manifest.json`). This is a PCI-relevant compliance
  artifact generated by construction, not written by hand.
- **Tamper-evident audit log.** `audit.capa` chains each log line to the
  previous one (`audit.capa`), so editing any earlier entry breaks every
  later chain value.
- **Deterministic, explainable risk.** `fraud.capa` returns a verdict
  plus the list of reasons that produced it (regulatory requirement: risk
  decisions must be justifiable).

### The honest caveats (do not deploy this)

- **The cryptography is FAKE.** Capa has no crypto in its standard
  library. `crypto_stub.capa` contains deliberately trivial placeholders
  (the "hash" is reversible, the "encryption" is not encryption). A real
  system must replace them with vetted primitives (Argon2id, AES-256-GCM,
  HMAC-SHA-256) behind a KMS. The showcase is about the data-flow
  discipline *around* the crypto, which Capa enforces, not the crypto
  itself, which it cannot do today.
- **The information-flow guarantee has known boundaries.** It tracks
  direct flows (values, parameters, interpolation, derived values,
  function results), aggregate literals (a `@secret` in a struct field,
  list, or tuple makes the whole aggregate `@secret`), and mutable
  containers (a secret `push`ed / `add`ed / `set` into a `List` / `Set` /
  `Map` taints the container, so a later read does not launder it). All
  of that is *intra-procedural*. What it does NOT do: (a) cross a
  function boundary implicitly, by design, a parameter that should carry
  a secret must be annotated `@secret` (explicit flow); and (b) offer
  per-field precision, the granularity is whole-aggregate / whole-
  container (a secret field taints the whole struct). Those are on the
  Capa roadmap (S2 follow-ups).
- **No real persistence, concurrency, or atomicity.** Idempotency is an
  in-memory set; the audit log is a read-modify-write file. Real
  transactional integrity, resilience, and scale are out of scope.
- **ML anomaly detection is out of scope.** Only deterministic,
  explainable rules, which is the part a regulator can audit line by
  line (and which the brief's explainability requirement favours).

## Layout

| File | Role | Capabilities |
|---|---|---|
| `domain.capa` | shared types; `@secret` on cardholder data | none |
| `crypto_stub.capa` | **FAKE** crypto placeholders | none |
| `mask.capa` | the audited cardholder-data `declassify` bridges (PCI disclosure points) | none |
| `validate.capa` | structural validation, PSD2 dynamic linking, idempotency | none |
| `auth.capa` | SCA / MFA (>= 2 factor classes), PSD2 exemptions, lockout | none |
| `fraud.capa` | deterministic, explainable risk / AML rules | **none (proven)** |
| `audit.capa` | tamper-evident, append-only logging of masked data | `Fs` |
| `main.capa` | the section-7 transaction lifecycle, wired together; the audited decision-output bridge (`print_decision`) | `Stdio`, `Fs` |
| `leaky_example.capa` | counter-example: the leak the compiler rejects | (does not build clean, on purpose) |

## Run it

```sh
# Type-check + information-flow check (clean: no leaks)
capa --check main.capa

# Run the three demo scenarios (approve / decline / tampered)
capa --run main.capa

# See the information-flow checker reject a raw-PAN leak
capa --check leaky_example.capa

# Regenerate the full CRA conformity pack (SBOMs, manifest, VEX, provenance)
./generate.sh
```

## The compliance story in one screen

```
$ capa --manifest main.capa | jq '.summary.declassification_sites'
5

$ capa --manifest main.capa | jq '.functions[] | select(.source_name=="assess")
                                   | .provably_excluded_capabilities'
[ "Clock", "Db", "Env", "Net", "Proc", "Random", "Unsafe", ... ]
```

The fraud engine *provably cannot exfiltrate*. The five disclosure
points are named, with reasons, in the SBOM: four cardholder-data
bridges in `mask.capa`, plus the decision-output bridge in `main.capa`
(an authorization decision is the pipeline's legitimate public output,
derived from transaction data and risk rules, never from the PAN or
CVV; it inherits the `@secret` label from the card at the analysis's
whole-value granularity, so the disclosure is declared and recorded
where the decision leaves the program). That is the artifact a CRA
/ PCI reviewer wants, produced by the compiler rather than asserted in a
PDF.

## CRA conformity pack

[`conformity/`](conformity/) is a worked EU Cyber Resilience Act
conformity pack for this program: the CycloneDX + SPDX SBOMs, the
capability manifest (with the information-flow `declassifications`), a
VEX document, and SLSA build provenance, all emitted by the compiler,
plus [`CONFORMITY.md`](conformity/CONFORMITY.md) mapping them to the
specific Annex I technical requirements they support. Regenerate it
with `./generate.sh`. It is a showcase, not a certification (see the
honesty boundary in `CONFORMITY.md`).

## Requirement mapping (brief -> mechanism)

| Brief requirement | How Capa carries it |
|---|---|
| 4.2 SCA / MFA | `auth.sca_satisfied` (>= 2 factor classes); secrets stay `@secret` |
| 4.3 RBAC / least privilege | capability parameters; SBOM `provably_excluded_capabilities` |
| 4.4 dynamic linking, idempotency | `validate.authorization_code` / `validate.claim` |
| 4.5 SCA exemptions | `auth.evaluate_exemption` (+ recorded reason) |
| 4.6 protect sensitive data | `@secret` labels + `mask.capa` declassify bridges |
| 4.7 explainable fraud / AML | `fraud.assess` returns signals + `fraud.explain` |
| 4.8 secure logging (no secrets in clear) | information-flow check at the `Fs.write` sink |
| Audit trail integrity | `audit.capa` hash chain |
| SBOM / conformity evidence | `capa --manifest` (`declassification_sites`, capability proof) |
