# CRA conformity pack: capa_paymentguard

A worked **EU Cyber Resilience Act** (Regulation (EU) 2024/2847)
conformity pack for the `capa_paymentguard` showcase. It bundles the
machine-readable artifacts the Capa compiler emits for this program
and maps them to the specific Annex I technical requirements they
support.

The point of this pack: the evidence below is **generated by the
compiler, by construction**, not asserted in prose. An assessor reads
the artifacts, not a policy document.

## Honesty boundary (read first)

This is a **language showcase**, not a certified product.

- The integrity / authentication crypto is REAL; confidentiality and
  the credential KDF are still stubs. HMAC-SHA256 (the audit hash
  chain, the PAN fingerprint, the dynamic-linking code) and the
  constant-time tag comparison come from the `capa_hash` dependency: a
  pure, zero-capabilities Capa library whose release is GPG-signed and
  SLSA-attested and verified by `capa install`. Those are genuine
  HMAC-SHA256 values. What stays FAKE in `crypto_stub.capa` is exactly
  what `capa_hash` does not cover: the credential **KDF** (`slow_hash`,
  which belongs to Argon2id, not a bare SHA) and the at-rest **cipher**
  (`encrypt`, which belongs to AES-256-GCM under a KMS data key). A
  real product still replaces those two with vetted primitives behind a
  KMS. The pack demonstrates the data-flow discipline *around* the
  crypto; the integrity half of the crypto is now real and verifiable.
- Capa is a programming language: it addresses the **technical** slice
  of Annex I that interacts with how a product is built and described.
  The **organisational** CRA obligations (vulnerability disclosure
  policy, incident notification, conformity assessment modules, update
  distribution) are the manufacturer's regardless of language, and are
  out of scope here.
- The information-flow guarantee is intra-procedural with
  whole-aggregate granularity (see the project README). It holds for
  the flows this program uses; it is not a whole-system proof.

## Artifacts in this pack

| File | What it is | Generated by |
|---|---|---|
| `manifest.json` | Per-function capability manifest + IFC declassifications | `capa --manifest main.capa` |
| `sbom.cyclonedx.json` | CycloneDX 1.5 SBOM with capability metadata | `capa --cyclonedx main.capa` |
| `sbom.spdx.json` | SPDX 2.3 SBOM | `capa --spdx main.capa` |
| `vex.cyclonedx.json` | VEX (exploitability statements) | `capa --vex main.capa` |
| `provenance.slsa.json` | SLSA v1 build provenance | `capa --provenance main.capa` |

Regenerate the whole pack with `./generate.sh` from the repo root. On a
re-diff against what ships here, the only expected divergence is the
build timestamps (`timestamp`, `created`, `annotationDate`, `startedOn`,
`finishedOn`), which are stamped at generation time. The substantive
content (capabilities, declassification sites, functions) is stable and
reproducible; `manifest.json` carries no timestamp and is byte-identical.

## Headline findings for this program

Read straight off `manifest.json`:

- **70 functions** (the program plus the verified `capa_hash`
  dependency it now pulls in), **0 crossing `Unsafe`** (no FFI /
  raw-pointer escape hatch anywhere in the program or its dependency).
- **6 declassification sites**, every secret-to-public disclosure in
  the program, each with a stated reason:

  | Function | Reason recorded in the SBOM |
  |---|---|
  | `mask_pan` | PCI DSS 3.4: display only the last four PAN digits |
  | `storable_credential` | credential stored only as a salted slow-hash, never in clear |
  | `storable_ciphertext` | PCI DSS 3.4: PAN stored only as AES ciphertext under a KMS data key |
  | `process` | PCI DSS 3.4: disclosing that a transaction's PAN matched the blocklist drives the public refusal decision |
  | `process` | PCI DSS 3.4: a matched PAN blocklist fingerprint is recorded in the fraud-alert log; it is a keyed one-way value, not the PAN |
  | `print_decision` | authorization decision is the pipeline's public output: approve/decline/review plus a rule-derived reason, no cardholder data |

  The first three (in `mask.capa`) disclose *derived forms of cardholder
  data* at rest or for display. The two in `process` are the
  PAN-blocklist sink: the `@secret` PAN fingerprint is matched, in
  constant time, against a public watchlist, and on a hit both the match
  result and the matched fingerprint are written to a fraud-alert file.
  The last (in `print_decision`) discloses the authorization *decision*,
  which carries no cardholder bytes.

  These bridges are **load-bearing**: `process` opts into `@strict_ifc`,
  so deleting either of its declassifies makes `capa --check main.capa`
  fail (a `@secret` value reaches `Fs.write`, or `Fs.write` runs under
  secret control flow). `mask.pan_fingerprint` deliberately keeps its
  result `@secret` so the disclosure decision is taken, and audited, at
  the sink where the value actually leaves the program.

- **Least privilege, proven**: the fraud / AML engine (`assess`,
  `risk_to_decision`) declares **no capabilities** and the manifest
  lists `Net` (and `Fs`, `Env`, `Clock`, `Db`, `Proc`) in its
  `provably_excluded_capabilities`. The risk engine *cannot*
  exfiltrate, by type.

Everything else in the program is a compile-time guarantee: a card
number (`@secret`) that reached a log line or a network call without
passing through one of the declassify bridges above would be a
build error.

## CRA Annex I mapping

Only the rows this program supplies concrete evidence for are listed.
See [`docs/cra.md`](https://github.com/nelsonduarte/capa-language/blob/main/docs/cra.md)
in the compiler repo for the full article-by-article treatment.

| CRA reference | Requirement (paraphrased) | Evidence in this pack |
|---|---|---|
| Annex I Part I (2)(b) | secure-by-default configuration | Every function defaults to zero capabilities; `manifest.json` shows each function's declared set is exactly what its signature carries. |
| Annex I Part I (2)(d) | protection from unauthorised access | Capabilities are unforgeable, flow only through parameters; `manifest.json` `declared_capabilities` per function is the access-control surface. |
| Annex I Part I (2)(e) | confidentiality of processed data | IFC: card data is `@secret`; the compiler forbids it reaching a public sink unmasked. `manifest.json` `declassifications` enumerates the 6 sanctioned disclosures with reasons. |
| Annex I Part I (2)(f) | integrity against manipulation | Each function's caps are derivable from its signature; an SBOM diff (`sbom.cyclonedx.json`) flags any dependency edit that widens a capability set. |
| Annex I Part I (2)(g) | data minimisation | Least authority is the language default; IFC additionally keeps secret data from flowing to sinks it does not need to reach. |
| Annex I Part I (2)(j) | limit attack surfaces / external interfaces | The declared capability set per function *is* the interface contract; `manifest.json` + the WIT spec under `capa --wasm --component` are the machine-readable surface. |
| Annex I Part I (2)(k) | reduce incident impact (exploitation mitigation) | Capability attenuation bounds blast radius (`fs.restrict_to`, enforced soundly cross-function on the Wasm backend); the fraud engine's empty capability set caps its reach. |
| Annex I Part II (1) | SBOM in a machine-readable format | `sbom.cyclonedx.json` (CycloneDX 1.5) and `sbom.spdx.json` (SPDX 2.3), each a strict superset of the NTIA minimum: not just what is included, but what each component can do. |

Out of scope for the language layer (the manufacturer's
responsibility): (2)(e) the *encryption* half, (2)(h) DoS resilience,
(2)(m) data erasure, and all of Annex I Part II beyond (1) (disclosure,
remediation timelines, update distribution).
